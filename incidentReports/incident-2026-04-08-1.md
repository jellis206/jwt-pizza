# Incident: 2026-04-08

## Summary

Between 17:50 and 18:08 UTC on April 8, 2026, pizza order fulfillment experienced 100% failure rate due to an outage at the JWT Pizza Factory. The event was detected by Grafana alerting rules monitoring pizza order failure rates and confirmed via application logs showing factory error responses. All users attempting to place pizza orders during this window received 500 errors. Menu browsing and authentication were unaffected.

## Detection

This incident was detected by the "Pizza Order Failures Detected" alert rule in Grafana, which monitors the `pizza_failures_total` metric for sustained increases. The alert fired after the failure count exceeded the threshold of 3 failures within a 5-minute window. Push notifications were delivered to the on-call engineer via ntfy.sh.

## Impact

For approximately 18 minutes between 17:50 UTC and 18:08 UTC on 04/08/2026, all pizza orders submitted to the JWT Pizza Factory failed. Users who attempted to order pizzas received HTTP 500 responses with the message "Failed to fulfill order at factory." Menu browsing, authentication, and session management remained fully operational.

## Response

After receiving the alert notification, the on-call engineer reviewed the Grafana dashboard at metrics.urjellis.com and observed:
1. A sustained spike in `pizza_failures_total` with zero successful orders
2. `pizza_revenue_total` flatlined to $0/min
3. Error log entries from the factory endpoint containing HTTP 500 responses

The traffic simulation logs on the EC2 metrics server confirmed every order was failing with the message "Failed to fulfill order at factory." Each failed response included a `reportUrl` field pointing to a Pizza Factory support endpoint.

## Root cause

The JWT Pizza Factory experienced an outage on its `/api/order` endpoint, causing all order fulfillment requests to return error responses. The factory's error response body included a diagnostic report URL (`reportUrl`) for the support team to investigate and resolve the issue.

## Resolution

The incident was resolved by accessing the diagnostic report URL returned in the factory's error responses:

```
https://pizza-factory.cs329.click/api/support/1915865e7b4247dcb1ee25258f765897/report/mhl589gl
```

Visiting this URL returned the confirmation: `{"message":"Problem resolved. Pizza is back on the menu!"}`. Subsequent pizza orders immediately succeeded, and all metrics returned to normal within one order cycle (~15 seconds).

## Prevention

- Maintain continuous traffic simulation to ensure factory outages are detected promptly via order failure metrics rather than waiting for user reports
- Keep alert rules with appropriate thresholds: failure count >3 in 5 minutes for order failures, with 1-minute pending periods for other metrics
- Ensure the ntfy.sh notification bridge is running and accessible from the Grafana Docker network

## Action items

1. Consider adding an automated check that extracts and logs the `reportUrl` prominently when factory failures are detected, reducing time-to-resolution
2. Add a Grafana dashboard annotation for factory outage events to aid post-incident review
3. Evaluate whether alert pending periods should be shortened for critical order failure alerts to reduce detection-to-notification latency
